How This Was Built — Step by Step

Every phase of this project is documented below — from the first SSH key to the final automated deployment.

Phase 1 — SSH Setup & Key Authentication

✅ Complete

Generated two SSH key pairs (personal + GitHub Actions), authorized both on the Namecheap cPanel server, configured the local SSH client, and verified a live connection into the remote server at premium340.

ssh-keygen ed25519 cPanel SSH Manager authorized_keys ~/.ssh/config Port 21098

Phase 2 — Application Build & GitHub Push

✅ Complete

Built a Node.js/Express application with REST API endpoints, a live dashboard frontend, and deployment history tracking. Initialized the GitHub repository and pushed the first commit to the main branch.

Express v4 REST API JSON storage npm init git push

Phase 3 — Server Environment Setup

✅ Complete

SSH into Namecheap server, install NVM and Node.js LTS without root access, install PM2 process manager, and clone the GitHub repository onto the server. Run the app manually to confirm it works before automation.

NVM install Node.js LTS PM2 git clone curl localhost:3000

Phase 4 — CI/CD Pipeline with GitHub Actions

✅ Complete

Write the GitHub Actions workflow YAML file that triggers on every push to main, SSHes into the server using stored secrets, pulls the latest code, installs dependencies, and restarts the app with PM2.

deploy.yml GitHub Secrets appleboy/ssh-action git pull pm2 restart

Phase 5 — Process Management & Persistence

✅ Complete

Configure PM2 ecosystem file for production settings, set up auto-restart on server reboot, verify crash recovery, and configure log management for the running application.

ecosystem.config.js pm2 save pm2 startup log management

Phase 6 — Testing, Verification & Rollback

✅ Complete

Run the full deploy verification test by bumping the version, pushing to GitHub, watching Actions run, and confirming the new version appears on the live site. Test the rollback procedure using git reset.

version bump test /api/info verify git reset rollback security checklist
📚 Concepts

DevOps & MLOps Explained

The two disciplines powering modern software and AI delivery pipelines.

⚙️

DevOps

DevOps is a set of practices that combines software development (Dev) and IT operations (Ops). The goal is to shorten the development lifecycle and deliver high-quality software continuously.

It eliminates the wall between developers who write code and operations teams who deploy it — making everyone responsible for delivery and reliability.

CI/CD Infrastructure as Code Monitoring Automation Collaboration
🤖

MLOps

MLOps (Machine Learning Operations) applies DevOps principles to machine learning workflows. It bridges the gap between data scientists building models and engineers deploying them to production.

Without MLOps, models often live only in notebooks — MLOps gets them into real applications and keeps them accurate over time.

Model Training Model Registry Feature Stores Drift Detection A/B Testing
🔄

CI/CD Pipeline

Continuous Integration means code changes are automatically tested when pushed. Continuous Deployment means passing code is automatically deployed to production — no manual steps.

This project implements a real CI/CD pipeline: push code → GitHub triggers → SSH into server → pull code → restart app. All automatic.

GitHub Actions Automated Testing Zero-Touch Deploy Rollback
🔐

SSH & Key Auth

Secure Shell (SSH) lets you securely access remote servers over an encrypted connection. Key-based authentication replaces passwords with cryptographic key pairs — far more secure and automatable.

This project uses two separate key pairs: one for manual access, one dedicated to GitHub Actions automation.

Ed25519 Keys authorized_keys GitHub Secrets No Passwords
🟢

PM2 Process Manager

PM2 keeps Node.js applications alive after SSH disconnects, auto-restarts on crashes, manages logs, and can start apps automatically on server reboot — all without root access.

Essential for running Node.js on shared hosting where you have no system service manager like systemd.

Process Persistence Auto-restart Log Management Zero Downtime
🌐

Shared Hosting vs VPS

Shared hosting means many users on one server — no root access, no custom ports, limited control. VPS gives you a dedicated virtual machine with full root access.

This project proves that a real CI/CD pipeline is achievable even on shared hosting — using NVM, PM2, and .htaccess proxy as workarounds.

NVM User-space install .htaccess proxy Port 21098
